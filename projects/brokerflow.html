Le problème

Un broker reçoit des dizaines de dossiers de crédit par semaine, souvent incomplets ou ambigus. Deux erreurs coûtent cher : accepter un dossier risqué ou refuser un bon client. Un score seul ne suffit pas : il faut une décision claire, justifiée et défendable.

La solution

  • Modèle ML calibré pour estimer le risque de défaut
  • Policy déterministe pour transformer le score en décision métier
  • Agents LLM pour lire la note libre, vérifier la cohérence et générer une explication lisible

Sortie : ACCEPTABLE, REVIEW, REQUEST_DOCUMENTS ou ESCALATE avec motifs et résumé naturel.

Pourquoi des agents ?

Les agents LLM n'ont pas le pouvoir de décider : ils informent, extraient les signaux, vérifient la cohérence et rédigent des explications. Si un agent échoue ou si Ollama est indisponible, le système bascule automatiquement en mode déterministe.

AgentCe qu'il faitCe qu'il ne fait pas
note_parserExtrait les signaux de la note libre (retards, contradictions, ambiguïtés, négations)Modifie le score
reviewerVérifie la cohérence entre données, note et documents — produit des alertes typéesPrend une décision seul
summary_writerReformule la décision finale en langage métierA une autorité sur la recommandation

Résultats mesurés

  • AUC modèle calibré : 0.728
  • Brier score : 0.150
  • Seuil opérationnel (F1 optimal) : 0.231
  • Précision alertes reviewer : 1.00
  • Rappel alertes reviewer : 1.00
  • F1 parser de notes : 0.89
  • Validité JSON sorties agents : 100 %

Structure du projet

src/
  api/         → FastAPI — routes /v1/score, /v1/review, /v2/score
  agents/      → note_parser, reviewer, summary_writer (LLM + fallback déterministe)
  models/      → entraînement, évaluation, scoring runtime
  rules/       → policy déterministe + vérifications de cohérence
  features/    → feature engineering
  eval/        → benchmark agents sur 50 cas annotés
  ui/          → interface Streamlit broker
notebooks/
  03 → EDA risque crédit
  04 → feature engineering
  05 → modèles baseline
  06 → calibration + explicabilité (SHAP)
  07 → évaluation agents
  09 → comparaison champion/challenger
  10 → test live broker (notebook interactif)
tests/         → 44 tests pytest
data/
  agent_eval_cases.json  → 50 cas annotés (standard / hard / adversarial)

Démarrage rapide

  1. Installer les dépendances : make setup
  2. Télécharger le modèle LLM local : ollama pull qwen2.5:3b-instruct
  3. Générer les données, entraîner, lancer l'API : python -m src.data.generate_synthetic_cases, make train, make run
  4. Interface broker (optionnel) : streamlit run src/ui/app.py
  5. Interface Gradio (optionnel) : make run-gradio

API disponible sur http://localhost:8000 · Documentation interactive : /docs
